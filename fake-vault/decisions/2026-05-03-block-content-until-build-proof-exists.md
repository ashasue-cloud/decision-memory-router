# Decision: Block Content Until Build Proof Exists

## Decision

We decided to block content until build proof exists.

## Why

the content system should show real work, not planned work.

## Options Considered

- Chosen: We decided to block content until build proof exists.
- Rejected: drafting belief posts before the prototype exists.

## Tradeoff Accepted

slower posting cadence in exchange for stronger credibility.

## What This Affects

- Content briefs about the router.
- GitHub proof claims.
- Any post that wants to describe a product milestone.

## Assumptions

- Build receipts create more trust than belief-only posts.
- Slower cadence is acceptable if proof quality improves.
- A public repo or concrete failure can count as evidence, but a plan cannot.

## Missing Data

- Whether the target audience rewards proof-heavy posts with saves or substantive replies.
- Whether a limited public-safe milestone is strong enough for a post without implying private reuse works.
